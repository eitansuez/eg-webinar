    Retries¶

    Retries are an example of how EG extends the Kubernetes Gateway API using Policy Attachments.

    ---
apiVersion: gateway.envoyproxy.io/v1alpha1
kind: BackendTrafficPolicy
metadata:
  name: httpbin-trafficpolicy
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: httpbin
    namespace: default
  retry:
    numRetries: 5
    perRetry:
      backOff:
        baseInterval: 100ms
        maxInterval: 10s
      timeout: 250ms
    retryOn:
      httpStatusCodes:
        - 500
      triggers:
        - connect-failure
        - retriable-status-codes

    kubectl apply -f retries/httpbin-policy.yaml

    Review the proxy's "stats"¶

    Specifically, the envoy_cluster_upstream_rq_retry metric:

    watch 'egctl x stats envoy-proxy -n envoy-gateway-system \
  -l gateway.envoyproxy.io/owning-gateway-name=eg \
  -l gateway.envoyproxy.io/owning-gateway-namespace=default \
  | grep "envoy_cluster_upstream_rq_retry{envoy_cluster_name=\"httproute/default/httpbin/rule/0\"}"'

    In another terminal, call a failing endpoint

    curl --insecure --head https://httpbin.esuez.org/status/500

    curl --insecure --head https://httpbin.esuez.org/status/500 \
  --resolve httpbin.esuez.org:443:$GATEWAY_IP

    Summary¶

    To configure retries, we had to resort to a BackingTrafficPolicy, an extension to the Gateway API. In contrast, compare with timeouts, which are configured directly on the HTTPRoute resource, since timeouts are a part of the Kubernetes Gateway API specification.
